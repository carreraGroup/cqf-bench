# Security Policy

## Reporting a vulnerability

**Email (primary):** [akastroulis@carrera.io](mailto:akastroulis@carrera.io)

Do **not** open a public GitHub issue for an undisclosed security vulnerability.

Include:

- Affected files or components
- Steps to reproduce
- Impact assessment (what an attacker could do)
- Suggested remediation, if you have one

### GitHub Security Advisories (optional)

**Private vulnerability reporting** is a separate feature from **GitHub Advanced
Security**. Advanced Security covers code/secret scanning (often org-licensed) and
is **not** where you configure how researchers contact you.

If the repository is **public** and the feature is enabled, reporters may also use
**Security → Advisories → Report a vulnerability**. Maintainers can always create a
**draft security advisory** under **Security → Advisories** when they have admin
access. Private repositories should use **email** until you go public.

When in doubt, use email.

## Supported versions

Security fixes are applied on the default branch (`master`). There is no separate
LTS policy yet for early releases.

## Response expectations

We aim to acknowledge reports within a few business days. Timing for fixes depends
on severity and complexity.
